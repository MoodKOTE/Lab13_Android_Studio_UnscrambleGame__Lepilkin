# Название проекта: UnscrambleGame

## Краткое описание
Это мобильное приложение-игра, разработанное для платформы Android. Игра предлагает пользователю увлекательный геймплей с интуитивно понятным интерфейсом. Основная цель проекта — демонстрация современных подходов к разработке под Android с использованием декларативного UI и архитектурных компонентов Jetpack.

## Технологии
Проект построен с использованием следующего стека технологий:
- **Kotlin** — основной язык программирования.
- **Jetpack Compose** — современный инструментарий для создания нативного UI.
- **ViewModel** — компонент для хранения и управления данными интерфейса с учетом жизненного цикла.
- **StateFlow** — потоковый API для наблюдения за изменениями состояния в реактивном стиле.

## Функциональность приложения
Приложение поддерживает следующие возможности:
- Запуск и остановка игрового процесса.
- Отображение текущего счета и прогресса в реальном времени.
- Адаптивный интерфейс, поддерживающий поворот экрана.
- Сохранение состояния игры во время конфигурационных изменений.
- Навигация между основными экранами (Меню, Игра, Настройки).

## Ответы на контрольные вопросы

## 1. Что такое ViewModel и зачем он нужен?

- **Объяснение своими словами:**  
  ViewModel — это класс, предназначенный для хранения данных, связанных с пользовательским интерфейсом. Он решает проблему разделения логики отображения и бизнес-логики. Главная задача ViewModel — пережить пересоздание активности (например, при повороте экрана), чтобы пользователь не терял введенные данные или прогресс.
- **Что происходит с данными при повороте экрана без ViewModel?**  
  Без ViewModel Activity уничтожается и создается заново при изменении конфигурации (поворот экрана). Все данные, хранящиеся в полях Activity или View, теряются, и интерфейс сбрасывается к начальному состоянию.

## 2. В чём разница между MutableStateFlow и StateFlow?

- **Зачем создавать два поля: `_uiState` и `uiState`?**  
  `_uiState` (приватное, изменяемое) используется внутри ViewModel для обновления данных. `uiState` (публичное, только для чтения) предоставляется UI для наблюдения. Это предотвращает случайное изменение состояния извне (из View/Compose).
- **Что такое "backing property pattern"?**  
  Это паттерн проектирования, при котором приватное свойство (backing property) хранит реальное значение и позволяет его менять, а публичное свойство возвращает только интерфейс для чтения. В Kotlin это часто реализуется через `private val _flow = MutableStateFlow(...)` и `val flow: StateFlow<...> = _flow`.

## 3. Почему в ViewModel нельзя хранить Context или View?

- **Что может произойти, если сохранить ссылку на Activity в ViewModel?**  
  ViewModel живет дольше, чем Activity (например, при повороте экрана создается новая Activity, а старая ждет уничтожения). Если ViewModel хранит ссылку на старую Activity, сборщик мусора не сможет удалить её из памяти. Это приводит к **утечке памяти (Memory Leak)**, что вызывает падение производительности и краш приложения.

## 4. Чем StateFlow отличается от `remember` и `rememberSaveable`?

- **Различия:**  
  - `StateFlow` — это поток данных, управляемый ViewModel, жизненный цикл которого привязан к ViewModel.
  - `remember` — хранит состояние только в рамках текущей композиции (исчезает при рекомпозиции, если не сохранено).
  - `rememberSaveable` — сохраняет состояние при пересоздании Activity (конфигурационные изменения), но не переживает процесс смерти приложения.
- **Когда лучше использовать каждый подход?**  
  - `StateFlow`: Для бизнес-логики и данных, которыми управляет ViewModel.
  - `remember`: Для временного UI-состояния (например, открыто ли текущее меню).
  - `rememberSaveable`: Для простого сохранения ввода пользователя (текст в поле) без использования ViewModel.

## 5. Что такое data class в Kotlin и зачем он используется для UI State?

- **Определение:**  
  `data class` — это класс, предназначенный в первую очередь для хранения данных.
- **Автоматически генерируемые методы:**  
  `equals()`, `hashCode()`, `toString()`, `copy()`, а также компонентные функции (`component1()`, `component2()`...).
- **Зачем нужна функция `copy()`?**  
  Она позволяет создавать новую копию объекта с измененными значениями отдельных полей. Это критически важно для реализации **неизменяемости (immutability)** состояния в UI. Мы не меняем старый объект, а создаем новый, что гарантирует предсказуемость работы StateFlow и Compose.

## 6. Как работает `collectAsState()` в Compose?

- **Механизм:**  
  Эта функция подписывается на `StateFlow` внутри композабла и преобразует его в `State<T>`, который понимает Compose.
- **Почему UI автоматически обновляется?**  
  Когда значение в `StateFlow` меняется, `collectAsState` получает уведомление и помечает текущую область композиции как "грязную". Compose запускает **рекомпозицию** только для тех элементов, которые зависят от изменившегося состояния, обновляя экран.

## 7. Зачем разделять код на пакеты `data` и `ui`?

- **Причина:**  
  Разделение позволяет изолировать логику работы с данными (модели, репозитории) от логики отображения (экраны, компоненты, ViewModel).
- **Принципы архитектуры:**  
  Это поддерживает принципы **Separation of Concerns (Разделение ответственности)** и **Clean Architecture**. Такой подход делает код более читаемым, тестируемым и легким в поддержке.

## 8. Что происходит с ViewModel, когда приложение полностью закрывается (process death)?

- **Сохраняются ли данные в этом случае?**  
  Нет. ViewModel хранит данные в оперативной памяти. При смерти процесса (убийство приложения системой или пользователем) память очищается, и ViewModel уничтожается безвозвратно.
- **Какие альтернативы есть для долговременного хранения?**  
  Для сохранения данных между сессиями необходимо использовать механизмы персистентного хранения:
  - **DataStore** (для настроек и простых данных).
  - **Room** (SQLite база данных для сложных структур).
  - **SavedStateHandle** (для сохранения состояния ViewModel при смерти процесса, если это поддерживается системой).
